Penjelasan Detail Fitur: G. Fitur Pendukung & Keamanan
Bagian ini membahas mekanisme di balik layar (backend) yang menjaga stabilitas gerbang masuk ujian serta fitur pemantauan aktual bagi pengawas di dalam ruangan. Kedua fitur ini vital namun operasinya sangat spesifik.

1. Gatekeeper & Autentikasi Login (Welcome.php)
Controller khusus ini memiliki spesialisasi untuk melindungi "Pintu Depan" aplikasi secara berlapis-lapis sebelum otentikasi ke database dilakukan.

Logika & Fungsionalitas:

Pemblokiran Browser Usang (index):
Segera setelah halaman diakses, kontroler membedah User Agent Browser milik pengakses. Jika HTTP Request terdeteksi secara historis berasal dari Internet Explorer, blokiran layar penuh (Access Denied) blokbrowser_view memotong proses selanjutnya demi mencegah kerusakan layout dan exploit karena IE tidak lagi didukung oleh HTML5 modern.
Safe Exam Browser Look (index):
Jika konfigurasi wajib pakai "Mobile Exam Browser" (cbt_mobile_lock_xambro="ya") aktif, ketika device mobile/tablet masuk, skrip meraba string / user agent header apakah memuat kode kunci ZYACBT. Jika gagal (Siswa memakai Google Chrome biasa), Controller me-lempar tampilan merah gembok lockmobile_view secara hardcoded.
Proteksi Joki & Multi-Login (login):
Saat form disubmit dan PIN/Username lolos validasi, sistem mengecek pengaturan proteksi_multilogin.
Jika "ya", maka baris data Siswa pada tebel cbt_user dicek, mungkinkah flag boolean user_login nya bernilai 1.
Nilai 1 merepresentasikan siswa ini sudah login di perangkat PC 1. Jika ia mencoba masuk kembali di PC 2, sistem langsung menggugurkan aksesnya dengan teguran keras "User terdeteksi masih Login".
Mekanisme ini me-reset menjadi nol 0 hanya ketika siswa memencet tombol Logout sah logout(), atau terjadi pemaksaan Reset Sesi oleh Admin di modul Manajemen Peserta.
Force-Close Ujian on Logout (logout):
Saat siswa menekan tombol Keluar Ujian secara wajar, aplikasi akan membaca konfigurasi cbt_tutuptes_logout. Jika diatur "ya", sistem mencari semua Log Ujian di cbt_tes_user untuk siswa tersebut yang berstatus 1 (Sedang Dikerjakan). Secara agresif sistem meng-eksekusi kueri tesuser_status=4 untuk mengakhiri nasib ujian itu tanpa kompromi, mengunci jawaban terakhir mereka.
2. Live Monitoring / Pengawas Ujian (manager/Tes_hasil_operator.php)
Sebagai pelengkap dari Laporan Hasil (yang lazimnya dipakai Guru seusai Ulangan), fitur ini didedikasikan teruntuk tugas Proktor / Pengawas Laboratorium untuk kelancaran jalannya Ujian di detik ujian tersebut sedang terjadi.

Logika & Fungsi:

Identifikasi Spesifik via Token Lokal (index):
Pengawas (Misalnya Pengawas Ruang A) tidak perlu pusing melihat data pengawas Ruang B. Controller ini memanfaatkan User_ID akun Pengawas A dengan menarik token tes apa terakhir kali dikeluarkan oleh tangan si Pengawas A.
Ini diwujudkan dengan query $this->cbt_tes_token_model->get_by_user_now($user_id).
Dasbor Real-Time Pengerjaan (get_datatable):
Datatable ini di-refresh secara dinamis untuk mengawasi langsung berdasarkan kumpulan siswa pengerja yang memakai spesifikasi Token bersangkutan.
Tidak hanya menyajikan skor/nilai sementara temp->nilai, kolom tersulit dan fungsional tertinggi ada pada pelaporan Kalkulasi Waktu Berjalan.
Jika status tesuser_status bernilai 1, sistem menggunakan DateTime::diff PHP. $tanggal_tes->modify('+'.$temp->tes_duration_time.' minutes') ditandingkan dengan jam dinding Server saat itu.
Menghasilkan dua kemungkinan String yang memandu keliling lab Pengawas:
"Berjalan (- 14 menit)" -> Siswa bersangkutan masih sisa waktu 14 menit.
"Selesai" -> Menit telah habis tanpa intervensi, atau siswa dengan sengaja menekan Hentikan Ujian (selesai dikerjakan).
